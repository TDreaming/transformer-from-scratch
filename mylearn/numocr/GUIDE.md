# NumOCR 设计与上手指南

这份文档面向两类读者：

- 想快速跑通项目的人
- 想看明白这套实现为什么这么设计的人

如果你只想先用起来，优先看“快速开始”。  
如果你想理解实现思路，再继续看“设计目标”“核心模块”“训练与预测流程”。

## 1. 项目目标

`numocr` 是一个小型的数字识别示例，目标不是追求最复杂的 OCR 系统，而是提供一个：

- 容易运行
- 容易阅读
- 容易扩展
- 能完整覆盖训练到预测闭环

的 PyTorch 入门项目。

它解决的问题是：输入一张 `28x28` 的数字图片，输出这个图片最可能是哪一个数字 `0-9`。

## 2. 为什么这样设计

这套实现有几个刻意的取舍。

### 2.1 不依赖外网下载训练数据

很多数字识别示例会直接用 MNIST。这样做当然没问题，但会带来几个现实问题：

- 首次运行需要下载数据
- 在网络受限环境下不方便
- 初学者不容易看出“数据到底长什么样、怎么来的”

所以这里采用了“程序动态生成数字图像”的方案：  
用类似七段数码管的方式生成 `0-9` 数字，再加入轻微位置偏移和随机噪声，让模型学习一个简单但完整的分类任务。

这样做的好处是：

- 打开项目就能训练
- 数据生成逻辑完全透明
- 很适合教学和代码阅读

代价是：

- 这不是通用手写 OCR
- 它更像一个“结构清晰的数字分类 Demo”

### 2.2 模型保持简单

这里没有上 ResNet，也没有引入 Transformer、Attention 或复杂数据增强，而是用了一个小型 CNN。

原因很简单：

- 对 `28x28` 数字图像，简单 CNN 已经足够
- 代码量更小，更利于初学者理解
- 更容易把注意力放在训练流程本身，而不是模型技巧上

### 2.3 保留训练与预测的完整链路

这个项目不是只写一个模型类，而是完整包含：

- 数据集
- 模型
- 训练脚本
- 权重保存
- 预测脚本
- 单元测试

这样你可以从“定义问题”一路看到“训练出模型并拿来预测”的全流程。

## 3. 快速开始

项目根目录下推荐使用虚拟环境：

```bash
python3 -m venv .venv
./.venv/bin/pip install -r mylearn/numocr/requirements.txt
```

### 3.1 跑测试

```bash
./.venv/bin/python -m unittest discover -s mylearn/numocr/tests -v
```

### 3.2 训练模型

```bash
./.venv/bin/python -m mylearn.numocr.train
```

如果你只是想快速试一下，可以缩小训练规模：

```bash
./.venv/bin/python -m mylearn.numocr.train --epochs 2 --train-size 2000 --val-size 400
```

### 3.3 预测一个合成数字

```bash
./.venv/bin/python -m mylearn.numocr.predict --digit 8
```

### 3.4 预测本地图片

```bash
./.venv/bin/python -m mylearn.numocr.predict --image path/to/digit.png
```

建议图片尽量满足这些条件：

- 单个数字居中
- 图像接近正方形
- 背景尽量干净
- 最好是黑底白字

如果是白底黑字，脚本会自动尝试反相处理。

## 5. 数据是怎么生成的

数据生成逻辑是这套实现里最值得读的部分之一。

### 5.1 基本思路

每个数字由一组“段”组成，例如：

- `1` 主要由右上和右下两段组成
- `8` 由七段全部组成

代码里用 `SEGMENTS_BY_DIGIT` 表示这种映射关系。  
生成图像时，会把对应的矩形段画到 `28x28` 画布上。

### 5.2 为什么不是固定模板

如果每个数字永远是同一个模板，模型会很快记住，训练任务就太死板了。  
因此生成时加入了几种随机性：

- 线段粗细变化
- 整体位置轻微平移
- 每个线段局部抖动
- 稀疏噪声
- 高斯噪声

这些扰动的作用是：

- 让同一个数字出现不同“写法”
- 避免模型只记忆一个固定模板
- 提高预测时对轻微偏移和噪声的鲁棒性

### 5.3 数据集为什么按需生成

`SyntheticDigitDataset` 不是预先把所有图片保存到磁盘，而是在 `__getitem__` 时动态生成。

这样做有几个优点：

- 节省磁盘空间
- 不需要管理额外数据文件
- 每个样本的生成逻辑可复现

这里通过固定随机种子保证同一个索引对应的样本是稳定的，便于调试和测试。

## 6. 模型为什么选 CNN

模型 `SimpleDigitCNN` 很短，但对这个任务已经足够。

结构分两部分：

- 特征提取
  - 两层卷积
  - 两层 ReLU
  - 两次最大池化
- 分类头
  - 展平
  - 全连接层
  - Dropout
  - 输出 10 类 logits

这样设计的考虑是：

- 卷积适合提取局部图像结构
- 池化可以逐步压缩空间尺寸
- 小模型训练快，适合教学和反复实验
- Dropout 可以稍微抑制过拟合

## 7. 训练流程是怎么工作的

训练脚本 `train.py` 的流程可以概括为下面几步：

1. 解析命令行参数
2. 检查参数是否合法
3. 选择设备
4. 构造训练集和验证集
5. 创建 DataLoader
6. 初始化模型、优化器、损失函数
7. 进入 epoch 循环
8. 每轮先训练，再验证
9. 保存验证集表现最好的模型

### 7.1 为什么同时有训练集和验证集

训练集用于更新参数。  
验证集用于观察模型泛化能力，并决定当前 checkpoint 是否值得保存。

这能避免只看训练精度造成的误判，例如：

- 训练集精度很高
- 但验证集表现一般

这种情况通常说明模型只是在记训练数据，而不是学到了更稳健的规律。

### 7.2 为什么保存最佳模型

不是最后一轮一定最好。  
所以代码会记录“验证集准确率最高时”的模型权重，而不是盲目保存最后一个 epoch 的结果。

## 8. 预测流程是怎么工作的

预测脚本支持两种输入：

- `--digit`
  - 先动态生成一个指定数字的样本
  - 再把它送进模型预测
- `--image`
  - 读取本地图片
  - 转灰度
  - 缩放到 `28x28`
  - 必要时做反相
  - 送入模型预测

最终输出：

- 预测类别
- Top-3 概率

这样设计的好处是：

- `--digit` 适合自测链路是否正常
- `--image` 适合接入真实输入

## 9. 我在实现时重点考虑了什么

这里把关键考虑点直接列出来，方便你以后自己写类似项目时复用。

### 9.1 先保证闭环，再追求复杂度

最重要的是先把下面这条链路跑通：

数据 -> 模型 -> 训练 -> 保存权重 -> 加载权重 -> 预测

如果这条链路不完整，模型写得再复杂也没有实际意义。

### 9.2 代码要能单独阅读

我尽量把职责拆开：

- 数据生成在 `data.py`
- 网络结构在 `model.py`
- 训练逻辑在 `train.py`
- 推理逻辑在 `predict.py`

这样阅读时不用在一个大文件里来回跳。

### 9.3 少依赖、好运行

依赖只保留最必要的几项：

- `torch`
- `pillow`
- `numpy`

其中：

- `torch` 是核心框架
- `pillow` 用于读取本地图片
- `numpy` 主要是补齐 PyTorch 在部分环境中的底层依赖告警

### 9.4 让错误尽量早暴露

我补了几类基础校验：

- 数据集样本数必须大于 0
- 当前实现只支持 `28x28`
- 训练参数如 `epochs`、`batch-size` 必须为正数

这类校验虽然简单，但能让错误在程序一开始就暴露，而不是训练到一半才发现。

### 9.5 测试不追求花哨，追求覆盖关键路径

测试没有去做“完全等于某个固定 loss 值”这种脆弱断言，而是覆盖更稳定的关键点：

- 生成图像尺寸和数值范围是否正确
- 数据集输出格式是否正确
- 模型输出维度是否正确
- 训练和评估是否返回合理指标
- checkpoint 能不能正确加载
- 预测结果的概率分布是否合理

## 10. 单元测试覆盖了什么

当前测试主要覆盖 6 类能力：

1. `render_digit()` 生成结果尺寸和值域正确
2. `SyntheticDigitDataset` 返回 `(1, 28, 28)` 图像和整数标签
3. 非法数据集参数能正确报错
4. 模型前向输出维度为 `(batch, 10)`
5. 单轮训练和评估能正常执行，并返回合理指标
6. 模型权重保存后可以重新加载并执行预测

这组测试的目标不是证明模型“训练得最好”，而是证明：

- 关键模块能协同工作
- 训练与预测链路没有断
- 后续重构时容易发现回归问题

## 11. 常见问题

### 11.1 为什么我预测真实手写数字效果一般

因为训练数据是“数码管风格”合成图像，而不是真实手写数字。  
这套模型当前更擅长识别与训练分布相近的输入。

如果你想识别真实手写数字，可以继续扩展：

- 使用 MNIST 或自定义手写数据集
- 增强预处理
- 增加更丰富的数据增强
- 适当升级模型结构

### 11.2 为什么训练结果每次略有差异

虽然代码设置了随机种子，但不同硬件后端和底层实现仍可能带来细微差异。  
这是深度学习实验里比较常见的现象。

### 11.3 为什么这里不直接做成通用 OCR

通用 OCR 涉及的问题要复杂得多，例如：

- 多字符检测
- 字符切分
- 不同字体和背景
- 倾斜、透视、模糊
- 文本行级别识别

这个项目刻意只聚焦“单个数字分类”，是为了把学习曲线压低。

## 12. 如果你想继续扩展

这里给几个很自然的下一步方向：

### 12.1 换成真实数据集

把 `SyntheticDigitDataset` 替换为：

- MNIST
- EMNIST
- 你自己收集的数字图片

### 12.2 增加可视化

你可以扩展一个脚本来：

- 随机展示训练样本
- 保存预测图片
- 把预测结果写到图上

### 12.3 支持批量预测

让 `predict.py` 支持一个目录输入，对整批图片做推理并输出结果表。

### 12.4 记录更多训练信息

例如保存：

- 每轮 loss / accuracy
- 最佳模型对应 epoch
- 混淆矩阵

### 12.5 升级模型

可以尝试：

- 更深的 CNN
- BatchNorm
- 更强的数据增强
- 更复杂的图像预处理

## 14. 一句话总结

`numocr` 本质上是一个“为了教学和快速上手而设计的 PyTorch 数字识别最小闭环项目”。

它不是为了追求最强模型，而是为了让你能清楚看到：

- 数据从哪里来
- 模型为什么这样写
- 训练过程怎么组织
- 预测是怎么落地的
- 测试如何保证这套代码不容易坏
